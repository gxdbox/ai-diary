1.2 编解码架构

2.1 输⼊表示：词是如何被 “ 数字化 ” 的？
模型⽆法直接处理⽂字，必须将其转化为向量。这个过程分为三步。
第⼀步， Tokenization （分词）
⾸先，将句⼦切分成基本单元（ token ）。
例如： “ 我喜欢学习 AI 。 ”→ 分词结果： [“ 我 ”, “ 喜欢 ”, “ 学习 ”, “AI”, “ 。 ”]
每个 token 会被映射到⼀个唯⼀的 ID ，如 “AI”→1024 。
第⼆步，词嵌⼊（ Word Embedding ）
每个 token ID 通过查表转换为⼀个⾼维向量（如 768 维），这个向量蕴含语义信息。例
如， “ 国王 ” 和 “ 王后 ” 的向量距离较近。
第三步，位置编码（ Positional Encoding ）
由于 Transformer 没有顺序处理机制，必须显式告诉模型 “ 这个词在第⼏位 ” 。于是引⼊位
置编码 ⸺ ⼀种基于正弦函数的固定模式，加到词向量上，使模型能感知词序。
举个⽣活化⽐喻：
想象你在听⼀场脱⼝秀，演员说了 10 句话。如果这些话被打乱播放，你会完全
看不懂笑点。位置编码就像给每句话贴上时间戳，让模型知道 “ 哪句先说，哪
句后说 ” 。

这是 Transformer 最⾰命性的设计，也是⼤模型理解上下⽂的关键。

⾃注意⼒机制（ Self-Attention ）是⼀种让每个词根据上下⽂来动态调整它的⾃身表示
（即向量）的机制。即对于每个词，会根据与上下⽂其他词的相关性来重新计算，以得

到新的向量表示。
这样能帮助神经⽹络模型更好地捕捉到词与词之间的动态紧密关系，从⽽实现更准确的
预测输出。简单说，就是让句⼦中的每个词去关注其他所有词，并根据相关性分配 “ 注意
⼒权重 ” 。
⽐如句⼦： “ 猫坐在垫⼦上，它很舒服。 ” 当模型处理 “ 它 ” 时，会⾃动计算它与 “ 猫 ”“ 垫⼦ ”
“舒服 ” 等词的相关性，发现 “ 它 ” 最可能指代 “ 猫 ” ，于是给予 “ 猫 ” 更⾼的注意⼒权重。

度的同时还能保持甚⾄提⾼准确度。

2.2.4 解码器的独特设计：掩码⾃注意⼒
在⾃注意⼒机制中，掩码（ Mask ）的作⽤是确保模型在处理序列数据时，只关注到当前
位置之前的元素，⽽不能看到当前位置之后的元素。这种机制对于保持模型的合理性和
预测的准确性⾄关重要。
解码器也有⾃注意⼒层，但它必须遵守⼀个规则：在⽣成第 t 个词时，只能看到前 t-1 个
词，不能偷看未来。为此， Transformer 在解码器的⾃注意⼒中加⼊了掩码（ Masking ）
⸺ 将未来的词屏蔽掉，确保⽣成过程是⾃回归的。
同时，解码器还有⼀个 “ 编码器 - 解码器注意⼒ ” 层，⽤于关注输⼊句⼦的关键信息（在对
话模型中作⽤较⼩）。
三、⽂本⽣成机制：从概率到连贯语⾔
现在我们已经知道模型如何 “ 理解 ” 输⼊，接下来要解决的问题是：它是如何 “ 写出 ” ⼀句

这个过程叫做⾃回归⽣成（ Autoregressive Generation ），即 “ ⼀个字⼀个字地猜 ” 。
3.1 ⽣成流程
⼤语⾔模型的⽣成机制可以概括为以下流程：输⼊  (Prompt) → Tokenization ( 词元化 ) →
Embeddings ( 向量化 ) → [ 循环处理块： Self-Attention ( 上下⽂理解 ) + Feed-Forward
Network ( 信息处理 )] → 输出⼀个概率分布  → Decoding/Sampling ( 选择⼀个词 ) → 将新
词加⼊输⼊序列  → 重复循环。
以提示词 “ 请写⼀⾸关于春天的诗 ” 为例，梳理以下⽣成的过程。
1. 编码上下⽂：模型将整个提示词输⼊，通过多层 Transformer 处理，⽣成⼀个深层语
义表示。
2. 预测第⼀个词：模型基于上下⽂，输出⼀个概率分布，表示接下来最可能出现的词是
什么。例如：
“春⻛ ” ： 30%
“花开 ” ： 25%
“阳光 ” ： 20%
……
3. 采样选词：根据某种策略（如贪婪搜索、随机采样等）选择⼀个词作为输出，⽐如选
了 “ 春⻛ ” 。
4. 循环迭代：将 “ 春⻛ ” 加⼊上下⽂，重新计算下⼀个词的概率分布，继续⽣成，直到遇
到结束符（ EOS ）或达到⻓度限制。
整个过程就像 “ ⾛迷宫 ” ，每⼀步都会根据当前环境选择下⼀个出⼝。
3.2 解码策略：决定⽣成⻛格的关键

=== 第 13 页 ===
为了让⽣成内容更有创造性和多样性，我们可以在概率分布中引⼊随机性。这⾥介绍两
种采样策略。
先来看 Top-K 采样。在所有词元中，只考虑概率最⾼的  K  个，然后在这  K  个词元中按照
它们的相对概率进⾏随机采样。优点是限制了采样范围，避免了选中那些⾮常不靠谱的
低概率词元。
另⼀种策略是 Top-P (Nucleus) 采样，这是⼀种更动态的策略。从概率最⾼的词元开始，
不断累加它们的概率，直到总和超过⼀个预设的阈值  P （例如  0.95 ）。然后，模型只从
这个概率总和超过  P  的核⼼词元集合（ Nucleus ）中进⾏采样。
它的优点是采样集合的⼤⼩是动态的。当模型⾮常确定下⼀个词时（例如 “I love” 之后很
可能是 “you” ），这个集合会很⼩；当模型不确定时（在⼀个开放式问题的开头），集合会
很⼤，允许更多探索。这通常被认为是效果最好的采样策略之⼀。
如表  4-1 所示，不同的采样策略会带来截然不同的输出⻛格。
实践建议：
在情感聊天机器⼈中，推荐使⽤  Top-p （如 0.9 ）  + 温度
（ temperature=0.7 ），既能保持⾃然流畅，⼜避免过于机械或胡⾔乱语。
3.3 温度（ Temperature ）的作⽤

=== 第 14 页 ===
温度  (Temperature) 采样，是⼀个调节参数。⾼温（ >1 ）   会让原始概率分布变得更平
缓，使得低概率词元更容易被选中，结果更具创造性、随机性，但也可能出现更多错误
和胡⾔乱语。低温（ <1 ）   则让概率分布更尖锐，⾼概率词元更容易被选中，结果更接近
贪⼼搜索，更保守、确定。 Temperature=0  就等同于贪⼼搜索。
温度参数控制了输出的 “ 随机性 ” 程度。
低温（如 0.1 ）：模型更保守，倾向于选择最⾼概率词  → 输出稳定但呆板
⾼温（如 1.5 ）：模型更⼤胆，低概率词也有机会被选中  → 输出有创意但可能出错
类⽐：
低温  = ⽼教授写论⽂，严谨但枯燥
⾼温  = 诗⼈即兴创作，惊艳但可能跑题

4.1 优化提示⼯程
优化提示⼯程的⾸要原则是清晰、具体并提供上下⽂。应使⽤明确的动作动词直接表达
需求，避免模糊指令，并详细说明输出的内容、结构、格式和质量要求。同时，解释任
务的⽬的、受众或限制原因，有助于模型更准确地理解意图，⽣成贴合实际需要的响
应。
进阶技巧包括使⽤示例、授权不确定性、引导推理过程和设定⻆⾊。通过提供⼀个或多
个输出范例（ one-shot/few-shot ），可以⾼效传达⻛格或格式要求；允许模型在信息不
⾜时承认 “ 不知道 ” ，可减少幻觉；加⼊ “ 请⼀步步思考 ” 等链式思维提示，能提升复杂任
务的表现；⽽指定⻆⾊（如 “ 作为⼼理咨询师 ” ）则可调整语⽓与专业深度。

=== 第 15 页 ===
对于复杂任务，应采⽤多步骤拆解与结构化输出策略。将⼤⽬标分解为若⼲⼩步骤依次
执⾏，并明确要求返回如  JSON 、表格等机器可读格式，不仅提⾼结果质量，也便于后
续处理。同时，清晰设定边界 ⸺ 说明哪些内容不应出现 ⸺ 与说明⽬标同样重要。
最后，提示⼯程是⼀个持续迭代的过程，需避免常⻅陷阱如过度复杂、隐含假设或⽭盾
指令。建议从简单提示开始，通过评估结果不断优化，建⽴有效提示库，并结合具体领
域（如代码、内容、数据分析）调整策略，从⽽系统性提升  AI 协作效率与输出质量。
4.2 合理管理上下⽂
在构建基于⼤语⾔模型的智能体或对话系统时，上下⽂管理已成为决定性能与可靠性的
关键环节。尽管现代模型⽀持数⼗万甚⾄百万  token 的上下⽂窗⼝，但 “ 注意⼒预算 ” 有
限、⻓上下⽂检索能⼒衰减、计算成本⾼昂等问题依然存在。因此，如何在有限资源下
⾼效组织和利⽤上下⽂信息，成为上下⽂⼯程（ Context Engineering ）的核⼼课题。
4.2.1 上下⽂裁剪：精准舍弃冗余信息
当对话历史或输⼊⽂本逼近模型  token 上限时，最直接的⽅法是裁剪（ Truncation ）。然
⽽，简单地截断开头或结尾往往会导致关键信息丢失。
优化做法包括以下⼏种。
优先保留最近⼏轮交互（通常包含最新意图）。
识别并保留包含任务⽬标、约束条件或⽤户身份的关键语句。
避免删除⼯具调⽤结果或错误反馈等对后续决策⾄关重要的内容。
裁剪不是粗暴删除，⽽是在 “ 保真 ” 与 “ 精简 ” 之间寻找平衡点。
4.2.2 滑动窗⼝：动态维护近期上下⽂
滑动窗⼝策略通过维护⼀个固定⻓度的上下⽂ “ 窗⼝ ” ，随对话推进不断更新内容。这种
⽅法特别适⽤于实时对话场景。

=== 第 16 页 ===
典型实现：
仅保留最近  N 轮对话（如最后 5 轮）。
结合时间戳或对话轮次编号，确保逻辑连贯性。
局限与应对：
可能丢失早期设定的⽬标（如 “ 帮我写⼀篇关于⽓候变化的科普⽂章 ” ）。
建议将初始任务描述单独缓存，并在每轮提示中显式重申。
滑动窗⼝虽简单，但需辅以任务锚点机制，防⽌ “ 记忆漂移 ” 。
4.2.3 摘要压缩：⽤⾼密度信息替代原始记录
当上下⽂过⻓时，可引⼊摘要压缩（ Summarization ）技术，将多轮对话或⻓⽂档提炼
为简洁语义表示。
应⽤场景示例如下。
将 10 轮客服对话总结为： “ ⽤户反馈订单 #12345 未收到，要求加急处理，偏好电话联
系。 ”
在代码  Agent 中，将之前的调试过程压缩为： “ 已尝试⽅案 A 失败，错误为 Timeout ；
当前聚焦于优化数据库连接池。 ”
实施建议：
使⽤专⻔的压缩提示词，要求模型保留决策依据、未解决问题和关键结论。
先追求⾼召回率（不漏信息），再迭代提升准确性（去冗余）。
压缩后的新上下⽂应作为 “ 记忆锚点 ” ，与最新交互共同构成完整状态。
摘要压缩本质上是⼀种有损但可控的信息编码，是突破上下⽂窗⼝限制的第⼀道防线。

=== 第 17 页 ===
4.2.4 关键信息提取：结构化存储核⼼要素
相⽐全⽂保留，更⾼效的⽅式是从上下⽂中提取结构化关键信息，如实体、意图、偏
好、状态变量等。
实现路径包括这样⼏种。
利⽤命名实体识别（ NER ）或⼩模型抽取时间、地点、⼈物、⾦额等。
将⽤户偏好（如 “ 只看经济舱 ”“ 避免周末出⾏ ” ）存⼊外部状态变量。
在每次调⽤  LLM 前，将这些结构化数据注⼊系统提示词。
这样做的优势是，极⼤节省  token 消耗，提升模型对约束条件的理解⼀致性，⽽且容易
与业务系统集成（如  CRM 、订单系统）。
这种 “ 语义蒸馏 ” 策略，让上下⽂从 “ ⽂本堆砌 ” ⾛向 “ 知识驱动 ” 。
4.3.5 分层记忆机制：短期 + ⻓期协同⼯作
为⽀持⻓时间、多阶段任务（如代码重构、研究报告撰写），需构建分层记忆架构。
短期记忆：当前会话的上下⽂，直接放⼊  prompt 。
⻓期记忆：通过向量数据库或⽂件系统持久化存储的历史笔记、⽤户画像、项⽬进展
等。
典型模式：
Agent 定期⽣成  NOTES.md ⽂件，记录待办事项、架构决策或实验结果。
下次启动时，通过检索相关笔记重新加载上下⽂。
⽀持 “ 渐进式信息披露 ”⸺Agent 按需查询记忆，⽽⾮⼀次性加载全部。
Anthropic 的  Claude Code 即采⽤此类策略。它不加载整个代码库，⽽是通过  grep 、
head 等⼯具动态检索所需⽂件，并将关键结论写⼊笔记供后续使⽤。

=== 第 18 页 ===
分层记忆不仅突破了  token 限制，更模拟了⼈类 “ 按需回忆 ” 的认知机制。
⽆论是裁剪、摘要、提取还是分层记忆，所有策略都指向同⼀个核⼼原则：在有限的注
意⼒预算内，最⼤化上下⽂的信息密度与相关性。随着模型能⼒增强，未来  Agent 或将
更⾃主地管理上下⽂ ⸺ ⾃动判断何时压缩、何时检索、何时遗忘。但在当下，深思熟

（ temperature ）参数过低，模型倾向于选择⾼概率词，缺乏探索能⼒；若同时缺乏重复
