3.2 基础对话链：让 AI“ 会聊天 ”
在  LangChain 中， LCEL （ LangChain Expression Language ）是构建处理流程的核⼼⽅
式，它通过直观的 “ 管道 ” （ | 操作符）将不同组件串联起来，形成可复⽤、可扩展的处理
链。chain = prompt | model | output_parser 正是  LCEL 最经典的基础链结
构，其核⼼是实现 “ 输⼊处理 → 模型调⽤ → 输出解析 ” 的完整流程。
组件化与管道流  LCEL 的设计理念是 “ 组件化 ”⸺ 将⼤语⾔模型交互的各个环节拆分为
独⽴组件（如提示词构建、模型调⽤、输出解析等），再通过  | 符号（类似  Unix 管道）
将这些组件按顺序连接，形成数据处理的 “ 流⽔线 ” 。
在  prompt → model → output_parser 这个链中，数据会按顺序流过每个组件：原始输
⼊先进⼊  prompt 组件，被格式化为模型可理解的提示词；格式化后的提示词再流⼊
model 组件，由⼤语⾔模型⽣成原始输出；模型的原始输出最后进⼊  output_parser 组
件，被解析为更易⽤的格式（如字符串、字典等）。  这种流程就像⼯⼚的⽣产线，每个
环节专注于⾃⼰的任务，最终输出成品。
各组件的⻆⾊与作⽤：
prompt ：负责将原始输⼊（如⽤户问题、上下⽂信息等）格式化为符合模型要求的
提示词结构。

=== 第 12 页 ===
model ：⼤语⾔模型（核⼼推理）  model 是链的核⼼，负责接收  prompt ⽣成的完整
提示词，进⾏推理并⽣成原始输出。
output_parser ：输出解析器（结果格式化）  output_parser 负责将模型的原始输出
转换为更易⽤的格式。
当调⽤  chain.invoke(inputs) 时，整个流程会⾃动触发  python 运⾏。假设输⼊为⽤户问
题， inputs = {“user_input”: “ 我最近总是失眠，该怎么调整？ ”} 。
执⾏步骤为：
1. prompt 接收  inputs ，⽣成完整提示词，再传给  model 。
2. model 接收提示词，调⽤⼤模型⽣成原始输出，随后传给  output_parser 。
3. output_parser 接收原始输出，解析为字符串，再返回最终结果。
