## 一、需求定义与场景拆解

### 1.1 核心目标

#### 为什么做（价值原点）

**核心痛点**：用户写完日记后无人倾诉，日记记录了情绪但没有后续反馈，用户缺乏情感宣泄后的回应。

**社会背景**（来自第02章）：
- 全球3亿抑郁症患者，心理咨询资源不足
- 孤独经济兴起，年轻人渴望"被倾听、被理解、被陪伴"
- 情感支持是人类基础且刚性的需求（马斯洛需求层次）

#### 做什么（边界范围）

**核心定位**：情感陪伴（非医疗级心理健康支持助手）
**目标用户**：松果日记用户

---

### 1.2 已明确的场景信息

#### 触发场景

| 场景 | 用户需求 | AI回应重点 |
|------|---------|-----------|
| 负面情绪日记 | 宣泄情绪、寻求安慰 | 共情回应 + 情感安慰 |
| 困惑/迷茫日记 | 理清思路、获得引导 | 引导反思 + 建议帮助 |

#### AI 回应形式（功能分层）

| 层级 | 功能 | MVP阶段 | 说明 |
|------|------|---------|------|
| 核心功能 | 共情回应、情绪接纳、上下文记忆 | ✅ 必须 | 基础情感陪伴能力 |
| 增强功能 | 长期记忆、情绪趋势分析、个性化人格 | 暂缓 | 后续迭代 |
| 安全边界 | 危机干预、敏感词过滤、隐私保护 | ✅ 必须 | 必须具备 |

---

### 1.3 设计原则（来自第02章）

**"支持而非替代"**：
- 明确告知用户"我是AI助手，不是心理咨询师"
- 设置安全护栏，高危情绪转介专业资源
- 引导用户回归现实连接，而非沉溺虚拟

**"缓冲带"理念**：
> 在用户自由倾诉与专业干预之间，设置一层结构化、低风险、有引导性的中间交互层

## 二、技术选型与架构设计

### 2.1 技术栈选择（基于第03章）

| 组件 | 选择 | 理由 |
|------|------|------|
| 后端框架 | FastAPI | 高性能、异步支持、类型安全 |
| 开发框架 | LangChain | 复杂Agent流程、社区活跃、生态完善 |
| 大模型 | 通义千问 API | 快速验证、国内合规、成本可控 |
| 短期记忆 | SQLite/MySQL | 当前会话消息存储 |
| 长期记忆 | ChromaDB | 向量数据库，跨会话语义检索 |
| 前端 | React | 现代化界面、组件化开发 |

**选型策略**：初期用API快速验证 → 成熟后考虑开源模型私有化部署

### 2.2 三大范式应用

#### Prompt 工程（核心）

```python
chain = prompt | model | output_parser  # LCEL链结构
```

**Prompt模板要素**：
- 角色定义：温暖、耐心、富有同理心的情感陪伴者
- 行为准则：先共情、后支持、避免说教
- 安全规则：危机关键词触发转介机制
- 对话示例：引导模型输出风格

参考 [PROMPT_TEMPLATE.md](PROMPT_TEMPLATE.md)

#### RAG（长期记忆）

**知识库内容**：用户历史日记数据
**检索策略**：
- 短期记忆：最近5-10条会话消息（MySQL）
- 长期记忆：语义相似的历史对话（ChromaDB）

**流程**：
```
用户输入 → 检索短期记忆 → 检索长期记忆 → 
拼接为prompt → 调用模型 → 保存到记忆库
```

#### Agent（可选扩展）

MVP阶段暂不实现复杂Agent能力，专注基础对话交互。


## 四、评估与优化

### 4.1 评估维度（来自第03章）

| 维度 | 方法 | 目标 |
|------|------|------|
| 共情能力 | 人工评分1-5分 | ≥4分 |
| 人格一致性 | 多轮对话检查 | 不漂移 |
| 安全性 | 注入测试用例 | 危机转介100%生效 |
| 用户满意度 | A/B测试 + 反馈收集 | NPS≥50 |

### 4.2 优化策略

- Prompt迭代：尝试不同角色设定（知心朋友 vs 专业倾听者）
- 记忆权重调整：近期记忆 > 远期记忆
- 反馈闭环：允许用户对回复打分👍👎
- 日志分析：统计常见情绪类型、中断率、会话时长
